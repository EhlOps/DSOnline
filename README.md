# DSOnline

Stream 1–8 isolated MelonDS Nintendo DS emulator instances to separate browser tabs using noVNC — no client software required. Designed to run on a Linux server or Oracle Cloud VM with Docker.

```
Docker Container
├── Display :1 → MelonDS (Player 1) → VNC :5901 → noVNC :6081 → nginx /player1
├── Display :2 → MelonDS (Player 2) → VNC :5902 → noVNC :6082 → nginx /player2
└── ... up to 8 players
```

Each player gets their own isolated display, VNC session, password, save data, and config.

---

## Requirements

- Ubuntu 22.04 VM (OCI free tier works — Ampere A1 with 2+ OCPUs recommended)
- Docker + Docker Compose
- Nginx
- Python 3.10+
- Ports **80/443** (nginx), **5901–590N** (VNC, optional), **6081–608N** (noVNC) open in your firewall / OCI Security List
- A domain name pointed at your VM's public IP (required for HTTPS)

## Quick Start

### 1. Clone the repo

```bash
git clone https://github.com/yourname/DSOnline.git
cd DSOnline
```

### 2. Add your ROMs

```bash
mkdir -p roms
cp your_game.nds roms/
```

ROMs are mounted read-only and shared between all players.

### 3. Run the setup wizard

```bash
python3 setup.py configure
```

The wizard prompts for:
- Number of players (1–8)
- Domain name or IP
- Per-player VNC passwords (or auto-generate)
- Resolution (720p recommended)
- Target architecture (auto-detect, ARM64, or x86\_64)

This writes `.env`, `docker-compose.yml`, `nginx/melonds-generated.conf`, and per-player `nginx/playerN.htpasswd` files.

### 4. Deploy

```bash
python3 setup.py deploy
```

This runs configure → start → nginx in one step. You can also run each stage separately:

```bash
python3 setup.py start   # build and launch the container
python3 setup.py nginx   # install and reload the nginx config
```

### 5. Open in browser

| Player | URL |
|--------|-----|
| Player 1 | `http://<your-domain>/player1/vnc.html` |
| Player 2 | `http://<your-domain>/player2/vnc.html` |
| ... | ... |

Each player is prompted for their VNC password on connect.

## Local Multiplayer (DS Wireless)

MelonDS emulates the Nintendo DS local wireless system. All instances run on the same host and communicate over loopback automatically.

1. Open the same ROM in each player window
2. In each instance, go to **System → Boot Firmware** to reach the DS home screen
3. Launch the game from the DS menu in each window
4. Navigate to the game's **Download Play** or **Wireless** multiplayer mode
5. One player hosts, the others join — MelonDS handles the rest

The instances discover each other automatically since they share a network namespace inside the container.

## Controls

Controls are captured from the keyboard/mouse of whoever is viewing each noVNC session. Each player's inputs are fully independent.

**Default MelonDS keybinds** (configurable per-instance via `Config → Input and Hotkeys`):

| DS Button | Default Key |
|-----------|-------------|
| A | X |
| B | Z |
| X | S |
| Y | A |
| L | Q |
| R | W |
| Start | Enter |
| Select | Right Shift |
| D-Pad | Arrow Keys |
| Touchscreen | Mouse click (bottom screen area) |

Keybind changes are saved to each player's isolated config directory and persist across restarts.

## Save Data

Save data and emulator config are stored in named Docker volumes, isolated per player:

| Player | Volume | Path inside container |
|--------|--------|-----------------------|
| Player 1 | `player1-data` | `/home/vncuser/.config/melonDS/player1` |
| Player 2 | `player2-data` | `/home/vncuser/.config/melonDS/player2` |
| ... | ... | ... |

Data persists across container restarts and image rebuilds. To back up a player's saves:

```bash
docker cp melonds-multiplayer:/home/vncuser/.config/melonDS/player1 ./backups/player1
```

To reset a single player's data:

```bash
docker compose down
docker volume rm dsonline_player1-data
docker compose up -d
```

To tear down the whole setup (optionally including save data):

```bash
python3 setup.py deconfigure
```

## Configuration

`.env` is generated by `setup.py configure`. You can also copy `.env.example` and edit it manually:

| Variable | Default | Description |
|----------|---------|-------------|
| `NUM_PLAYERS` | `2` | Number of active players (1–8) |
| `PLAYER1_VNC_PASSWORD` | `player1` | Browser password for Player 1 |
| `PLAYER2_VNC_PASSWORD` | `player2` | Browser password for Player 2 |
| `VNC_GEOMETRY` | `1280x720` | Streamed desktop resolution (WxH) |
| `DOMAIN` | `localhost` | Domain or IP used when generating the nginx config |
| `PLATFORM` | auto | Docker build platform (`linux/arm64`, `linux/amd64`) |

> `.env` is gitignored. Never commit real passwords.

## HTTPS with Certbot (Recommended)

Running over plain HTTP means your VNC passwords and screen streams are unencrypted. Use Certbot to get a free Let's Encrypt certificate.

**Prerequisites before running Certbot:**
- A domain name with an A record pointing to your VM's public IP (bare IPs are not supported)
- Ports **80 and 443** open in your firewall / OCI Security List
- The app already deployed and nginx already running (`python3 setup.py deploy`)

Certbot works by temporarily serving a challenge file over HTTP on port 80. If port 80 is blocked or nginx isn't running, the challenge will fail.

### 1. Open ports 80 and 443 in your firewall

In your OCI Security List (or equivalent), add ingress rules for **TCP port 80** and **TCP port 443** if they aren't already open.

### 2. Deploy the app first

Certbot needs an existing nginx config to modify. If you haven't already:

```bash
python3 setup.py deploy
```

Confirm nginx is running and serving your domain over HTTP before continuing:

```bash
curl -I http://yourdomain.com/player1/vnc.html
# Should return 401 (auth required), not a connection error
```

### 3. Install Certbot

```bash
sudo apt install -y certbot python3-certbot-nginx
```

### 4. Obtain a certificate

```bash
sudo certbot --nginx -d yourdomain.com
```

When prompted, choose option **2 (Redirect)** to automatically redirect all HTTP traffic to HTTPS.

Certbot will:
1. Obtain a certificate from Let's Encrypt via an HTTP challenge on port 80
2. Edit `/etc/nginx/sites-available/melonds` in place to add SSL directives
3. Add a redirect block for port 80 → 443
4. Reload nginx

Your WebSocket proxy headers (`Upgrade`, `Connection`) are preserved — noVNC will continue working over HTTPS/WSS.

### 5. Verify the resulting nginx config

After Certbot runs, your nginx config at `/etc/nginx/sites-available/melonds` will look similar to this:

```nginx
server {
    server_name yourdomain.com;

    location /player1/ {
        auth_basic "Player 1";
        auth_basic_user_file /etc/nginx/player1.htpasswd;
        proxy_pass http://localhost:6081/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
    }

    # ... additional player blocks ...

    listen 443 ssl;
    ssl_certificate /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;
}

server {
    listen 80;
    server_name yourdomain.com;
    return 301 https://$host$request_uri;
}
```

Check it looks correct, then test and reload:

```bash
sudo nginx -t && sudo systemctl reload nginx
```

### 6. Verify auto-renewal

Certbot installs a systemd timer that renews certificates automatically before they expire. Confirm it's active and do a dry run:

```bash
sudo systemctl status certbot.timer
sudo certbot renew --dry-run
```

### 7. Access over HTTPS

| Player | URL |
|--------|-----|
| Player 1 | `https://yourdomain.com/player1/vnc.html` |
| Player 2 | `https://yourdomain.com/player2/vnc.html` |

### Troubleshooting Certbot

**"Connection refused" or challenge fails**
Port 80 must be open and nginx must be running. Check `sudo systemctl status nginx` and your firewall rules.

**"No names were found in your configuration files"**
Make sure your nginx config has a `server_name` set to your actual domain (not `localhost`). Re-run `python3 setup.py configure` with the correct domain and then `python3 setup.py nginx`, then retry Certbot.

**Certificate issued but noVNC shows a blank page or WebSocket error**
Verify the `proxy_set_header Upgrade` and `Connection "upgrade"` lines are still present after Certbot edited the config. Certbot generally preserves these, but check with `sudo nginx -T | grep -A2 Upgrade`.

**Re-running after reconfiguring players**
If you add or remove players and re-run `python3 setup.py nginx`, it overwrites `/etc/nginx/sites-available/melonds` with a fresh HTTP-only config, removing Certbot's SSL additions. After any `setup.py nginx` run, re-run Certbot to re-apply SSL:

```bash
sudo certbot --nginx -d yourdomain.com
```

## Security Notes

- Each player's noVNC path is protected by HTTP basic auth (htpasswd) — always run behind HTTPS in production
- noVNC ports (`6081–608N`) are bound to `127.0.0.1` only and proxied through nginx — do not expose them publicly
- Once HTTPS is set up you can remove the noVNC port mappings from `docker-compose.yml`
- For direct VNC client access, use an SSH tunnel rather than opening VNC ports:

```bash
ssh -L 5901:localhost:5901 ubuntu@<your-vm-ip>
# Then connect your VNC client to localhost:5901
```

## Troubleshooting

**Black screen in browser**
Check container logs:
```bash
docker compose logs -f
```

**Password prompt not appearing**
Make sure your browser URL ends in `/vnc.html`, not just the root path.

**Players see the same screen**
Confirm you're using the correct URLs (`/player1/` vs `/player2/`). Each path proxies to a different noVNC port.

**Save data missing after restart**
Volumes persist by default. The `-v` flag in `docker compose down -v` deletes volumes permanently. Use `docker compose down` (no `-v`) to preserve saves.

**Starting fresh**
```bash
python3 setup.py deconfigure
```
Stops the container, removes `.env`, the nginx config, and resets `docker-compose.yml` to the 2-player default. Optionally deletes all save data volumes when prompted.
